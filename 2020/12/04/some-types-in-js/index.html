JavaScript 中的一些类型小知识

String

String 用于表示文本大小，其最大长度是 2^53 - 1；

该长度，一般在开发中是够用的。

另外，在 Safari 浏览器中，input 标签中的文本内容如果过长，会出现内容消失的 BUG。

String 的意思并不是「字符串」，而是字符串的 UTF16 编码；

一些字符串的方法和属性如 charAt、charCodeAt、length ，这些针对的的都是 UTF16 编码。

所以，字符串的最大长度，实际上是受字符串的编码长度影响的。换句话说，在 JavaScript 中，字符串的长度并不是我们视觉所看到的字数的长度。

但是，在实际开发中，绝大多数的字符串长度都和我们视觉所看到的字符数量长度相等，如下面代码中，str 长度就是等于视觉看到 3 个字母的数量。

const str = 'abc';
str.length; // 3

这里就要提一下字符串的国际标准。在现行的字符串国际标准下，字符是以 Unicode 的方式表示的。每一个 Unicode 的码点表示一个字符。理论上，Unicode 的范围是无限的。

另外，UTF 是 Unicode 的一个编码实现方式，规定了码点在计算机中的表示方式，常见的有 UTF8、UTF16 等。

Unicode 的码点通常用「U + 四个或五个数字」来表示，其中这里的「数字」是十六进制的码点值。0 ～ 63536（U+000 ～ U+FFFF）的码点被称为基本字符区域（BMP）。

BMP（基本字符区域）域涵盖了我们最常见的字符，这些个字符可以用 UTF16 来定长表示的，也就是说，所有在基本区域的字符的 UTF16 的编码长度都是和字符数相等的，这也就解释了为什么前面 str.length 得到的长度和我们视觉所看到的长度一致。

JavaScript 字符串把每个 UTF16 单元当作一个字符串来处理，这也带来一个问题，就是处理非 BMP 的字符的时候，你要额外注意长度的处理。如：

// 这里说的问题，即如果超出了该范围，则 JS 中获取该字符的长度可能是 2，比如 一些 emoji。
const str = '👋';
console.log(str.length); // 2

// 这样会导致一些问题，如通过 charAt(0) 访问该字符的时候，实际上只能访问该 emoji 表情中的一部分.

之所以这么设计，是因为 JavaScript 继承自 Java，最新的标准中是这么解释的：这样的设计是为了“性能和尽可能实现起来简单”，因为现实中很少使用到 BMP 之外的字符。

JavaScript 中的字符串永远无法变更

在 JavaScript 中，一旦字符串构造出来，无法用任何方式改变字符串的内容，所以字符串具有值类型的特征。如：

var str = 'hello';

// 你无法使用该方法去变更 hello 这个字符串的内容，注意，我这里说的是 hello 这个字符串，不是 str 这个变量
str[0] = 'Lucy, h';
// 依旧输出 hello
console.log(str); 

// 当然，你可能会说可以这么干：
str = 'Lucy, hello';

// 可以这么干，只是对 str 这个变量进行了重新赋值，并不是对字符串内容做修改。

Number

Number 类型表示我们通常意义上的「数字」，这个数字大致对应数学中的有理数，当然，在计算机中，我们有一定的精度限制。

JavaScript 中的 Number 类型基本符合 IEEE 754-2008 规定的双精度浮点数规则，但是 JavaScript 为了表达几个额外的场景语言（比如不让除以 0 出错，而引入的无穷大的概念。）

  • NaN，该值占用了 9007199254740990 个特殊数字，用来表示 NaN ，这原本是符合 IEEE 规则的数字；（这也可以解释为什么 NaN !== NaN）
  • Infinity 无穷大
  • -Infinity 负无穷大

处理 +0、-0 以及 Infinity、-Infinity 的关系

在 Javascript 中，会有 +0 和 -0 的存在，在加法运算中他们没有区别，但是在除法的场合要特别注意，不要忘记检测 -0，避免得到无穷大的情况，而区分 +0 和 -0 的方式，可以通过如下方式去检测：

// 通过检测 1/x 是 Infinity 还是 -Infinity。
const num = 0;
1 / 0 === Infinity;
1 / -0 === -Infinity;

根据双精度浮点数的定义，Number 类型中有效的整数范围是 -0x1fffffffffffff（13个f）至 0x1fffffffffffff ，所以 Number 无法精确表示此范围外的整数。

小数在转化为二进制时，通过小数部分不停乘2取整，知道消暑部分为0，这种方式导致一些小数乘2永远做不到小数部分为0，只能取一定精度的值表示，进而导致精度误差的出现。

事实上，摆脱编程语言的限制，仅仅从浮点数的角度来看，浮点数是可以比较的，只不过由于精度的问题可能导致一些在计算中出现一些问题，所以 Js 才规定浮点数不可以进行比较。

对于在 JavaScript 中，检测浮点数的正确的方式是：

// Number.EPSILON 的意义在于，该常量是 JavaScript 中能够表示的最小精度值，当所计算的结果误差小于这个精度，则可以理解为没有误差
Math.asb(0.1 + 0.2 - 0.3) <= Number.EPSILON

Symbol

  • Symbol 是 ES6 引入的新类型，他是一些非字符串对象的 key 的集合，在 ES6 规范中，整个对象系统被 Symbol 重塑。
  • 一些标准中提到的 Symbol，可以在全局的 Symbol 函数的属性中找到。例如：我们可以使用 Symbol.iterator 来自定义 for..of 在对象上的行为
    var obj = new Object();
obj[Symbol.iterator] = function() {
  var v = 0;
  return {
    next: function() {
      return { value: v++, done: v > 10 }
    }
  }
}

for (var v of obj) {
  console.log(v);
}

这些 Symbol 形式的标准，构成了语言的一类接口形式，它允许编写与语言更加紧密的 API。

Object

在 JavaScript 中，对象的定义是“属性的集合”，属性分为数据属性和访问器属性，二者都是 key-value 的结构，key 可以是字符串或者 Symbol 类型。

事实上，JavaScript 中的 「类」仅仅是运行时对象的一个私有属性，而 JavaScript 中是无法自定义类型的。

我们需要区分基本类型和对象类型。在基本类型中，都在对象类型有“侵袭：“

  • Number；
  • String；
  • Boolean；
  • Symbol；

所以，我们必须认识到 3 和 new Number(3) 是完全不同的值，他们一个是 Number 类型，一个是对象类型；

Number、String、Boolean 这三个构造器是两用的，当跟 new 搭配时，他们产生对象，当直接调用时，他们表示强制类型转换

Symbol 函数比较特殊，直接调用 new 会报错，但他仍然是 Symbol 对象的构造器。

JavaScript 语言设计上试图模糊对象和基本类型之间的关系，比如我们日常代码可以直接在基本类型上调用对象的一些属性或方法：

"abc".charAt(0);

点(.)运算符为我们提供了「装箱操作」，他会根据基础类型构造一个临时对象，使得我们能够在基础类型上调用对象的方法和属性。

类型转换

String To Number

字符串数字的类型转换，存在一个语法结构，类型转换支持十进制、二进制、八进制和十六进制，比如：

  • 30
  • 0b111；
  • 0o13；
  • 0xFF

此外，JavaScript 支持的字符串语法还包括正负号科学计数法，可以使用大写或者小写的 e 来表示：

  • 1e3；
  • -1e - 2

需要注意的是，parseInt 和 parseFloat 并不使用这个转换，所以支持的语法跟这里不尽相同。

在不传入第二个参数的情况下，parseInt 只支持 16 进制前缀 “0x”，而且会忽略非数字字符，也不支持科学计数法。

在一些古老的浏览器环境中，parseInf 还支持 0 开头的数字作为 8 进制前缀，这是很多错误的来源，所以在任何情况下，都建议传入 parseInt 的第二个参数，而 parseFloat 则直接把原字符串作为十进制来解析，不会引入任何其他的进制。

Number To String

在较小的范围内，数字到字符串的转换是完全符合基本直觉的，即 1.toString() 返回 '1' 。但是当数值较大较小的时候，toStirng 返回的结果可能就是科学计数法的字符串形式。

这个是由 JavaScript 内部的相关算法来决定的，算法细节繁多，简单理解就是为了保证所产生的字符串不会过长。

装箱转换

每一种基本类型 Number、String、Boolean、Symbol 在对象中都有对应的类，所谓装箱转换，就是把基本类型转化为对应的对对象，他是类型转换中一种相当重要的种类。

前面提到，全局的 Symbol 对象无法使用 new 来调用，但是我们可以利用装箱机制来得到一个 Symbol 对象，我们可以利用一个函数的 call 方法来强迫产生装箱。

我们可以定义一个函数，函数里面只有 return this，然后我们调用函数的 call 方法到一个 Symbol 类型的值上，这样就会产生一个 symbolObject。

var symbolObject = (function() { return this; }).call(Symbol('a'));

console.log(typeof symbolObject); // object
console.log(symbolObject instanceof Symbol); // true
console.log(symbolObject.constructor == Symbol); // true

上面方法过于繁琐，我们可以直接使用内置的 Object 函数，显示的调用装箱能力：

var symbolObject = Object(Symbol('a'));
console.log(typeof symbolObject); // object
console.log(symbolObject instanceof Symbol); // true
console.log(symbolObject.constructor == Symbol); // true

每一类的装箱对象皆有私有的 class 属性，这些属性可以用 Objet.prototype.toString 获取：

var symbolObject = Object(Symbol('a'));

Object.prototype.toString.call(symbolObject); // [object Symbol]

这是为什么我们可以看到有些情况可以用该方法去判断其他的数据类型。

之所以可以这么识别，另一方面也是因为在JS 中，没有任何方法可以更改私有的 Class 属性，因此，Object.prototype.toString 是可以准确识别对象对应的基本类型的方法，他比 instanceof 更加准确。

需要注意的是，call 本身会产生装箱操作，所以需要配合 typeof 来区分基本来行还是对象类型。

拆箱转换

前面提到了装箱转换，对应的，也会有拆箱转换。

在 JavaScript 中，规定了一个 ToPrimitive 函数，他是对象类型到基本类型的转换，即拆箱转换。

对象到 String 和 Number 的转换都遵循「先拆箱再转换」的规则，通过拆箱转换，把对象变成基本类型，再从基本类型转换为对应的 String 或者 Number。

这也就对应了我们平时所见到的一些怪异的面试题。

拆箱转换会尝试调用 valueOf 和 toString 来获得拆箱后的基本类型，如果 valueOf 和 toString 都不存在，或者没有返回基本类型，则会产生类型错误 TypeError。

var o = {
  valueOf: () => { console.log('valueof'); return {} },
  toString: () => { console.log('toString'); return {} }  
}

o * 2;
// valueOf
// toString 
// 拆箱转换失败，抛出 TypeError

String(o);
// 先执行 toString
// valueOf
// TypeError

同时需要注意的是，vlaueOf 和 toString 具体哪个先执行，取决于你的后续操作

另外，我们也可以显示的设置对象的 @@toPrimitive Symbol 来覆盖原有的行为（就是前面的拆箱转换过程）。

// 依旧使用前面的对象 o
o[Symbol.toPrimitive] = () => { console.log('toPromitive'); return 'hello' }

o + '';
// toPromitive
// hello
