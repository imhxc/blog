forEach遍历Array引发的一个知识点

发表于 2018-02-27 |
Array对象大家都已经熟悉了, 但是肯定会多多少少有一些忽略的知识点。

大家可以阅读另外一篇文章：js对象深如探析，里面有清晰的结论。或者建议大家直接阅读《你不知道的JavaScript》第三章:对象。

在一次敲代码中, 对一个数组调用了forEach迭代器, 发现一个诡异的现象: 自定义的key无法被迭代出来。

但是, 当我使用for...in的时候, key等于user的值却可以打印出来。

当然，for...in循环出来的是key，这并不影响我们获取value。

到现在, 我们就可以提出一个疑问： 为什么forEach迭代器无法迭代出自定义的值, 而for…in却可以？

要想知道这个问题, 我们必须先了解Array对象中的元素, 或者说什么样的数据才可以算是arr中的元素?

在ECMA官方文档中发现了这样的介绍：

Array objects give special treatment to a certain class of property names. A property name P (in the form of a String value) is an array index if and only if ToString(ToUint32(P)) is equal to P and ToUint32(P) is not equal to 2^32−1. A property whose property name is an array index is also called an element.

翻译成通俗的话就是:

只有属性名称的数值不等于2^32^−1(或者说在0到2^32^-2之间)才会被视为数组的元素

文档地址

我们又知道forEach只迭代数组的元素, 而我们自定义的key无法满足规范, 所以严格上不能成为数组元素, 因此无法被迭代出来。

而for...in则不然, 它负责的是遍历所有可枚举的属性, 职责功能不同, 所以造就了结果不同。

在文章开始我们说了 ‘自定义的key无法被迭代出来’ 的说法是不严谨的， 相信到这里应该也可以明白为什么不严谨了。其实根本原因是因为属性名被转换后的数值必须不等于2^32^−1才算是数组元素。

总结

通过上面的分析, 我们可以得到下面几个结论:

  • 严格意义上, 并不是所有在数组中出现的元素都属于数组;
  • 元素的属性值在经过转换后需要不等于2^32^−1才算是数组元素;
  • forEach只遍历属于该数组的元素;
  • map也只会便利属于概述组的元素;
  • JSON.stringify也只转换属于该数组的元素;
  • for...in等都只负责遍历可枚举的属性，只要该属性可枚举， 就遍历该属性。

大家可以试一下其他的一些迭代方法， 比如reduce、for...of等等。

需要注意的是, 对于JSON.stringify方法也会忽略不属于数组的元素, 也可以理解为其无法表示这种东西, 所以忽略。
