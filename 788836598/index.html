前言

我们经常使用过 ReactDOM.render(<App />, document.body)，可是你是否想过调用 render 的时候其所经历的整个过程，现在我们就通过源码来分析。

其主要代码在 /react-dom/src/client/ReactDOM.js 文件里。

文中的代码只会粘贴一些核心部分，完整的可以参考官方的源码文件。

React 最主要的三个更新方式

在这之前，我们先来了解下 React 最主要的三个更新方式：

  • ReactDom.render || hydrate：render 和 hydrate 本质都一样，唯一不同的是后者为服务端渲染，后面会提到。
  • setState
  • forceUpdate

render 步骤

先来看下调用 render 的一个大致步骤：

  • 创建 ReactRoot；
  • 创建 FiberRoot 和 RootFiber；
  • 创建一个更新，创建一个，就会进入一个更新调度阶段。

ReactDOM.render 源码分析

一般的，我们使用 React 会将组件挂在到某个节点上：

ReactDOM.render(<App />, document.body)

对于上面这段代码，我们来看 React 源码中的定义：

const ReactDOM: Object = {
  createPortal, // Portal 组件会调用，也就是 React 传送门，如 Dialog 组件会用到
  findDOMNode() {...},
  hydrate() {...}, // 和 render 本质是一样的，唯一的区别在于是否会调和服用 container 里面的 HTML 节点，主要在服务端渲染的时候会使用该 API
  render(
    element: React$Element<any>, // React 组件
    container: DOMContainer, // 容器，如：document.getElementId('app')
    callback: ?Function, // 回调函数
  ) {
    // 返回调用 legacyRenderSubtreeIntoContainer 后的结果
    return legacyRenderSubtreeIntoContainer(
      null,
      element,
      container,
      false, // 这里后续 forceHydrate 参数的值，一般处于服务端渲染的时候会为 true
      callback,
    );
  }
  ...
};

可以发现， ReactDOM 其实就是一个常量对象，里面包含了一些属性、方法，其中就有 render。

对于 render 和 hydrate 的唯一区别就是在调用 legacyRenderSubtreeIntoContainer 方法的时候传递的第四个参数不同，render传递的是 false，而 hydrate 传递的是 true。

对于 render 方法，接收三个参数，对于前面两个我们都见过，但是实际上它还可以接收一个回调函数作为第三个参数。

render 最终将这些接收的参数传递并调用 legacyRenderSubtreeIntoContainer 方法，然后返回该方法执行的结果。

legacyRenderSubtreeIntoContainer：

function legacyRenderSubtreeIntoContainer(
  parentComponent: ?React$Component<any, any>, // 父组件
  children: ReactNodeList,
  container: DOMContainer,
  forceHydrate: boolean,
  callback: ?Function,
) {
  // TODO: Ensure all entry points contain this check
  invariant(
    isValidContainer(container),
    'Target container is not a DOM element.',
  );
  let root: Root = (container._reactRootContainer: any);
  if (!root) {
    // 初始化 root
    root = container._reactRootContainer = legacyCreateRootFromDOMContainer(
      container,
      forceHydrate,
    );
    // 如果有 callback 会进行封装调用一下
    if (typeof callback === 'function') {
      ...
    }
    // 这里会涉及到 React 中的一个批量更新的概念
    DOMRenderer.unbatchedUpdates(() => {
      if (parentComponent != null) {
        // 这里基本不会走，因为在当前情况 parentComponent 永远为 null
        ...
      } else {
        root.render(children, callback);
      }
    });
  } else {
    ...
  }
  return DOMRenderer.getPublicRootInstance(root._internalRoot);
}

首先会先获取 root：

let root: Root = (container._reactRootContainer: any);

不过正常情况，我们第一次渲染的时候，container 作为一个 DOM 对象，一定是不会有 _reactRootContainer 属性的。

所以，紧接着，React 对 root 不存在，也就是第一次渲染的情况下做了判断处理：

if (!root) {
  // Initial mount
  // 这里调用 legacyCreateRootFromDOMContainer 创建了一个 root
  root = container._reactRootContainer = legacyCreateRootFromDOMContainer(
    container,
    forceHydrate,
  );
  ...
  // Initial mount should not be batched.
  DOMRenderer.unbatchedUpdates(() => {
    if (parentComponent != null) {
      root.legacy_renderSubtreeIntoContainer(
        parentComponent,
        children,
        callback,
      );
    } else {
      root.render(children, callback);
    }
  });
}

在 root 不存在的情况，React 会调用 legacyCreateRootFromDOMContainer 来为其初始化一个 root。

legacyCreateRootFromDOMContainer：

legacyCreateRootFromDOMContainer 方法最终也会返回一个 ReactRoot 实例：

function legacyCreateRootFromDOMContainer(
  container: DOMContainer,
  forceHydrate: boolean,
): Root {
  // 判断是不是服务端渲染
  const shouldHydrate = forceHydrate || shouldHydrateDueToLegacyHeuristic(container);
  // 如果不是服务端渲染，则通过循环删除 container 子节点
  if (!shouldHydrate) {
    let warned = false;
    let rootSibling;
    while ((rootSibling = container.lastChild)) {
      container.removeChild(rootSibling);
    }
  }
  // Legacy roots are not async by default.
  const isConcurrent = false;
  // 返回一个 ReactRoot 实例
  return new ReactRoot(container, isConcurrent, shouldHydrate);
}

对于 forceHydrate 前面也提到了，就是用来区分是不是服务端渲染，如果使得话，就会调用 shouldHydrateDueToLegacyHeuristic 方法：

function shouldHydrateDueToLegacyHeuristic(container) {
  const rootElement = getReactRootElementInContainer(container);
  return !!(
    // 是不是存在
    rootElement &&
    // 是否是普通的 element type
    rootElement.nodeType === ELEMENT_NODE && 
    // 是否有 ROOT_ATTRIBUTE_NAME，这里 ROOT_ATTRIBUTE_NAME 值为 data-reactroot，服务端渲染会在 root 节点添加该属性，以此来标识该应用是由服务端渲染的
    rootElement.hasAttribute(ROOT_ATTRIBUTE_NAME) 
  );
}

该方法通过判断来决定是否需要进行服务端渲染。

然后 legacyCreateRootFromDOMContainer，中，通过 shouldHydrate 来标识是否是服务端渲染，如果不是的话，会通过一个循环来删除 container 节点下的所有子节点：

if (!shouldHydrate) {
  let warned = false;
  let rootSibling;
  while ((rootSibling = container.lastChild)) {
    container.removeChild(rootSibling);
  }
}

最终，legacyCreateRootFromDOMContainer 返回一个实例：

return new ReactRoot(container, isConcurrent, shouldHydrate);

对于 ReactRoot，你可以查看其方法定义，会发现并没有做太多的操作，主要还是调用了 DOMRender 的一个方法来生成 root。 而 DOMRender 引自 react-reconciler ：import * as DOMRenderer from 'react-reconciler/inline.dom';，react-reconciler 是 React 中非常重要的模块，它会去操作和平台无关的一些节点的调和、以及任务调度等。 这里我们可以简单的了解下，调用 DOMRenderer.createContainer 后，最终会返回一个 FiberRoot，也就进入到了我们前面提到的第二部。

回到最开始的地方，也就是 ReactDOM 对象中的 render 属性方法，接着往下看，其会继续调用 root.render(children, callback);,对于 root.legacy_renderSubtreeIntoContainer 在这里是不会调用的，因为 parentComponent 永远不存在：

DOMRenderer.unbatchedUpdates(() => {
  if (parentComponent != null) {
    root.legacy_renderSubtreeIntoContainer(
      parentComponent,
      children,
      callback,
    );
  } else {
    root.render(children, callback);
  }
});

ReactBatch.prototype.render：

root.render(children, callback); 所调用的就是 ReactBatch.prototype.render：

ReactBatch.prototype.render = function(children: ReactNodeList) {
  invariant(
    this._defer,
    'batch.render: Cannot render a batch that already committed.',
  );
  this._hasChildren = true;
  this._children = children;
  const internalRoot = this._root._internalRoot;
  const expirationTime = this._expirationTime;
  const work = new ReactWork();
  DOMRenderer.updateContainerAtExpirationTime(
    children,
    internalRoot,
    null,
    expirationTime,
    work._onCommit,
  );
  return work;
};

对于 updateContainerAtExpirationTime 这设计到 React 另外一个重要的概念，即调度更新、优先级，这个需要我们单独的去了解下。

这里简单提一下，简单的看下该方法的源码，会发现其会调用 scheduleRootUpdate，这个非常重要，也就是要开始调度了。 其首先会生成一个 update，因为在 React 中，不管是 setState 还是 ReactDOM.render 造成的更新，都会生成一个 update 对象，并且赋值给 Fiber.updateQueue，最终进入 scheduleWork。
